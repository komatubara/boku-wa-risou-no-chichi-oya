# コーディング実装コマンド

あなたはゲーム開発者のアシスタントです。仕様書に基づいて未実装タスクを検出し、ユーザーと方針を確認した上で `game.html` にコードを実装します。

## 進め方のルール

1. **一度に実装するのは1〜2機能**に留める（小さな単位で着実に進める）
2. **単一HTMLファイル構成を維持**する（`game.html` に全て収める）
3. **既存コードのパターン・命名規則に従う**（タイル定義、state管理、関数構成等）
4. **仕様書に書かれていない機能は勝手に追加しない**
5. ユーザーが $ARGUMENTS で特定機能を指定した場合は、現状分析をスキップしてその機能の実装に集中する

## 手順

### Step 1: 現状分析

以下のファイルを読み込む：
- `廃校の怪_仕様書.md` — 開発ロードマップ、未決定事項
- `game.html` — 現在の実装状態
- `scripts.md` — ゲーム内テキスト（存在する場合）

読み込んだ情報から、以下を整理して表示する：
- **実装済み**: ロードマップのうち、既にコードに反映されている項目（チェック済み含む）
- **未実装**: まだコードに反映されていない項目
- **未決定**: 仕様書の「未決定事項」セクションに該当し、実装前に設計が必要な項目

$ARGUMENTS が指定されている場合はこの分析をスキップし、指定された機能に関連する仕様・コードだけを確認する。

### Step 2: 優先度提案

未実装タスクから、次に実装すべきタスクを **2〜3個** 提案する。提案には以下を含める：
- タスク名（仕様書のロードマップ項目に対応）
- 提案理由（依存関係、実装の容易さ、ゲーム体験への影響）
- 実装の概要（どの関数を追加/変更するか、おおよその行数）

### Step 3: ディスカッション

ユーザーに実装対象と方針を確認する（**最大3問**）：
- どのタスクを実装するか
- 実装方針で判断が必要な点（複数のアプローチがある場合）
- 仕様書の「未決定事項」に該当する場合、その場で決定してもらう

質問には**選択肢を2〜4個提示**する（自由記述も常に歓迎する旨を伝える）。

### Step 4: 実装

確認が取れたら、`game.html` を編集してコードを実装する：
- 既存のコード構造（定数定義 → タイル型 → 部屋定義 → state → 入力 → 関数 → 描画）に従って適切な位置にコードを追加する
- 変更箇所が多い場合は、変更の概要を先に説明してから編集する
- 実装後、変更内容のサマリーを表示する

### Step 5: 動作確認ガイド

実装後、ユーザーに以下を提示する：
- ブラウザで確認すべき操作手順（具体的なキー入力やシナリオ）
- 正常動作の期待結果
- 注意して見るべきポイント（エッジケース等）

### Step 6: 仕様書更新

実装が完了したら：
- `廃校の怪_仕様書.md` のロードマップの該当チェックボックスを更新（`[ ]` → `[x]`）
- 未決定事項がStep 3で決定された場合、仕様書の該当箇所も更新
- 更新内容をユーザーに報告

## 注意事項

- `game.html` は単一ファイルで全てのゲームロジックを含んでいる。外部ファイルへの分割は行わない
- タイル型を追加する場合は `T` オブジェクトと、必要に応じて `SOLID` セットを更新する
- 部屋を追加する場合は `rooms` オブジェクトに追加し、ドア遷移の整合性を確認する
- イベントテキストは `events` オブジェクトに `"x,y"` キーで追加する
- `state` に新しいプロパティを追加する場合は、初期化とリセット処理の両方を更新する
- PLiCy での動作を考慮し、外部リソース（画像、音声URL等）への依存は最小限にする
