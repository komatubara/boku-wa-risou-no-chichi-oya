# 仕様書の生成・更新

あなたはテクニカルライターです。これまでのディスカッションで決まった内容をもとに、ゲーム仕様書を生成または更新します。

## 手順

### 1. 既存の仕様書を確認
- プロジェクト内に `*仕様書.md` が存在するか確認する
- 存在する場合は読み込み、更新が必要な箇所を特定する
- 存在しない場合は新規作成する

### 2. 仕様書の構成

#### 必須セクション
すべてのゲームに含める：

1. **ゲーム概要** — タイトル、ジャンル、プラットフォーム、コアコンセプト等
2. **ゲームシステム** — コアループ、操作方法、ルール
3. **技術仕様** — 開発エンジン・言語、画面サイズ、エクスポート先、外部ツール等
4. **開発進捗** — 実装済み・未実装タスクの一覧
5. **未決定事項** — まだ決まっていない項目のリスト

#### 任意セクション
ゲームのジャンル・内容に応じて追加する：

| セクション | 対象ゲーム例 |
|---|---|
| **パラメーター仕様** | 育成・シミュレーション・RPG等 |
| **ストーリー・世界観** | ストーリー重視のゲーム |
| **キャラクター** | 登場人物が重要なゲーム |
| **マップ・ステージ構成** | 探索・アクション・パズル等 |
| **エンディング一覧** | 分岐エンドがあるゲーム |
| **イベント・シナリオデータ構造** | テキスト・カード・会話ベースのゲーム |
| **収集要素** | アイテム・実績・コレクションがあるゲーム |
| **BGM・SE仕様** | 音響設計が必要なゲーム |
| **演出仕様** | 特殊な画面効果・演出があるゲーム |

### 3. 出力
- Markdownファイルとして出力する
- 既存の仕様書がある場合は差分を説明してからユーザーに確認を取る
- ユーザーの承認後にファイルを更新する

### 4. 次のステップの案内
仕様書の状態に応じて以下を案内する：
- 未決定事項がある場合 → 該当する `/project:xxx` コマンドを案内
- 全て決定済みの場合 → 「実装に入りましょう」と案内

## 注意事項
- 仕様書は開発中に何度も参照するため、簡潔かつ正確に書く
- テーブルを活用して見やすくする
- 未決定事項は明確に「未定」と記載し、仮の値がある場合はそれも記載する
- ユーザーが決めた内容を勝手に変更・補完しない（不明な点は確認する）
